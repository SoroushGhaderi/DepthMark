1. Extract Bronze Loading Service

Strong In-process

Files: scripts/bronze/load_clickhouse.py (1312 lines), src/services/

graph LR subgraph "BEFORE" A["load_clickhouse.py\n1312 lines"] --> B["DataFrame processing"] A --> C["Schema validation"] A --> D["DLQ fallback"] A --> E["ClickHouse insertion"] A --> F["Contract checks"] end subgraph "AFTER" G["load_clickhouse.py\nthin CLI wrapper"] --> H["BronzeService"] H --> I["match_processor"] H --> J["clickhouse_client"] H --> K["layer_contracts"] H --> L["dlq"] end

Before

  • 1312-line monolithic script with no service delegation
  • DataFrame processing, schema validation, DLQ fallback, ClickHouse insertion all inline
  • Only layer without a src/services/ counterpart
  • Cannot unit test workflow without invoking the full script

After

  • BronzeService owns workflow coordination, contract checks, DLQ, and summary
  • Script becomes a thin CLI wrapper (like Silver/Gold loaders)
  • Consistent service pattern across all three layers
  • Testable through the service interface without subprocess invocation

Problem: Bronze is the only layer where all workflow logic lives in the script, violating ADR 0010's pattern and making the 1312-line script untestable in isolation.

Solution: Extract a BronzeService following the same pattern as SilverService and GoldService. The script becomes a thin CLI wrapper.

Wins: Locality (workflow logic in one place), leverage (tests cross the service interface), consistency (all three layers follow the same script→service pattern).

ADR: Aligns with ADR 0010. No conflict.

2. Unify Configuration Systems

Strong In-process

Files: config/base.py, config/fotmob.py, config/settings.py

graph LR subgraph "BEFORE" A["config.yaml"] --> B["FotMobConfig"] C[".env"] --> B C --> D["Settings singleton"] B -. "duplicated" .-> D end subgraph "AFTER" E[".env + config.yaml"] --> F["Settings\nsingle source"] F --> G["FotMob config"] F --> H["ClickHouse config"] F --> I["Telegram config"] end

Before

  • Two independent config systems: YAML-based FotMobConfig and pydantic Settings
  • Duplicated settings (logging, proxy, metrics) with no reconciliation
  • Module-level side effects: load_dotenv() and Settings() at import time
  • Callers must know which system owns which setting

After

  • Single Settings class with FotMob-specific nested config
  • One source of truth for all configuration
  • No import-time side effects; explicit initialization
  • Clean testability with injected config

Problem: Two parallel config systems with duplicated settings, different override mechanisms, and import-time side effects create confusion and fragility.

Solution: Consolidate into a single pydantic-settings Settings class with nested config groups. Remove the YAML-based FotMobConfig or make it read from Settings.

Wins: Locality (one place to find any setting), leverage (tests inject one config object), interface clarity (callers don't need to know which system owns what).

3. Introduce Test Infrastructure

Strong Local-substitutable

Files: tests/ (absent), pytest.ini, pyproject.toml

graph TD subgraph "BEFORE" A["pytest.ini"] -. "points to" .-> B["tests/ directory\nDOES NOT EXIST"] C["pyproject.toml\n[tool.pytest]"] -. "points to" .-> B D["Zero test files"] --> E["Zero test coverage"] end subgraph "AFTER" F["tests/unit/"] --> G["date_utils, validation\nlayer_contracts, metrics"] H["tests/integration/"] --> I["SilverService, GoldService\nwith mock ClickHouse"] J["conftest.py"] --> K["shared fixtures\nmock clients, sample data"] end

Before

  • Zero test files. Tests deleted in commit afc17fe.
  • pytest.ini configured for non-existent tests/ dir
  • No CI/CD, no automated quality gates
  • Only validation: manual --dry-run modes

After

  • tests/unit/ for pure logic (date_utils, validation, layer_contracts)
  • tests/integration/ with mock ClickHouse for services
  • conftest.py with shared fixtures (mock clients, sample data)
  • Tests cross service interfaces, not implementation details

Problem: Zero test coverage. All validation is manual. No automated quality gates exist.

Solution: Create tests/ with unit tests for pure logic and integration tests with mock ClickHouse for services. Start with the most testable modules (date_utils, validation, layer_contracts) and work outward.

Wins: Leverage (one test suite catches regressions everywhere), locality (failures pinpoint the broken module), interface discipline (tests enforce module contracts).

4. ClickHouse Client Factory

Worth exploring Remote but owned

Files: src/storage/clickhouse_client.py, 8+ scripts with repeated setup

graph LR subgraph "BEFORE" A1["silver/load_clickhouse.py"] --> B1["10-line setup block"] A2["gold/load_clickhouse.py"] --> B2["10-line setup block"] A3["bronze/drop_clickhouse.py"] --> B3["10-line setup block"] A4["quality/check_*.py"] --> B4["10-line setup block"] end subgraph "AFTER" C1["any script"] --> D["create_clickhouse_client()\none function call"] D --> E["ClickHouseClient"] E --> F["connect / try / finally / disconnect"] end

Before

  • Same 10-line client instantiation pattern repeated in 8+ scripts
  • Same connect/try/finally/disconnect boilerplate everywhere
  • clickhouse_setup_common.py has its own connect_clickhouse() with extra logic other scripts don't need

After

  • create_clickhouse_client() factory or context manager
  • Scripts call one function instead of repeating setup
  • Connection pooling and cleanup in one place
  • Tests can inject a mock client at the seam

Problem: ClickHouse client setup boilerplate is copy-pasted across 8+ scripts, creating maintenance friction and inconsistency.

Solution: Add a create_clickhouse_client() factory function or a context manager to clickhouse_client.py. Scripts call one function instead of 10 lines of setup.

Wins: Locality (connection logic in one place), leverage (all scripts benefit from fixes), testability (factory returns a mockable seam).

5. Consolidate Drop Scripts

Worth exploring In-process

Files: scripts/bronze/drop_clickhouse.py, scripts/silver/drop_clickhouse.py

graph LR subgraph "BEFORE" A["bronze/drop_clickhouse.py\n113 lines"] --> B["_find_tables()"] A --> C["parse_args()"] A --> D["main()"] E["silver/drop_clickhouse.py\n113 lines"] --> F["_find_tables()"] E --> G["parse_args()"] E --> H["main()"] end subgraph "AFTER" I["scripts/drop_clickhouse.py\n--layer bronze|silver|gold"] --> J["drop_layer_tables()\nshared function"] end

Before

  • Two near-identical 113-line scripts differing only in default database name
  • Same _find_tables(), parse_args(), main() copy-pasted
  • Bug fixes must be applied to both files

After

  • Single scripts/drop_clickhouse.py --layer bronze|silver|gold
  • Or keep both as thin wrappers calling a shared function
  • One file to maintain, one place for bug fixes

Problem: Bronze and silver drop scripts are near-identical 113-line copies, creating duplication and maintenance risk.

Solution: Consolidate into a single script with a --layer argument, or extract a shared drop_layer_tables() function that both scripts call.

Wins: Locality (one implementation), leverage (bug fixes apply everywhere), simplicity (less code to maintain).

6. Telegram Notification Service

Worth exploring True external

Files: src/services/telegram/, 5+ scripts with repeated notification blocks

graph LR subgraph "BEFORE" A1["silver/load_clickhouse.py"] --> B1["TelegramClient()\nrender_and_send()"] A2["gold/load_clickhouse.py"] --> B2["TelegramClient()\nrender_and_send()"] A3["bronze/scrape_fotmob.py"] --> B3["TelegramClient()\nrender_and_send()"] A4["health_check.py\n(utils layer)"] -.->|"leaks into services"| B4["TelegramClient"] end subgraph "AFTER" C1["any script"] --> D["send_layer_alert()\nconvenience function"] D --> E["TelegramClient"] F["health_check.py"] --> G["optional callback\nno service import"] end

Before

  • Same TelegramClient() + render_and_send() pattern repeated in 5+ scripts
  • health_check.py (utils) imports Telegram service, creating circular dependency
  • Each script constructs LayerAlertData inline

After

  • send_layer_alert(layer, success, ...) convenience function
  • Health checks use a callback or optional notifier, not direct import
  • Scripts call one function instead of constructing dataclasses

Problem: Telegram notification boilerplate is duplicated across 5+ scripts, and health_check.py imports the Telegram service creating a utils→services dependency leak.

Solution: Add a convenience function like send_layer_alert() to the telegram module. Fix health_check.py to accept an optional notifier callback instead of importing Telegram directly.

Wins: Locality (notification logic in one place), leverage (all scripts benefit), interface cleanliness (health_check.py no longer depends on services).

7. Remove Empty Placeholder Packages

Speculative In-process

Files: processors/silver/, processors/gold/, storage/silver/, storage/gold/

Before

  • Four empty __init__.py files with only docstrings
  • Suggests Python processors/storage exist for Silver/Gold (they don't)
  • Misleads contributors about where Silver/Gold logic lives

After

  • Remove empty packages or add a README explaining they are intentionally empty
  • Silver/Gold logic lives in clickhouse/ SQL, not Python
  • Clearer codebase navigation for new contributors

Problem: Empty placeholder packages (processors/silver/, storage/gold/, etc.) create misleading expectations about where Silver/Gold logic lives.

Solution: Remove the empty packages or add a README.md in each explaining they are intentionally empty because Silver/Gold transformations live in ClickHouse SQL (per ADR 0007).

Wins: Clarity (new contributors find logic where it actually lives), reduced noise (fewer empty directories to navigate).

Top Recommendation

Start with Candidate 1: Extract Bronze Loading Service.

  • Follows an established pattern (Silver and Gold already have services)
  • Resolves the largest inconsistency (Bronze is the only layer without service delegation)
  • Unlocks testability for the heaviest script (1312 lines)
  • Aligns with ADR 0010 and requires no architectural decisions

Once the Bronze service exists, the test infrastructure (Candidate 3) becomes much easier to build because services provide clean test seams.
