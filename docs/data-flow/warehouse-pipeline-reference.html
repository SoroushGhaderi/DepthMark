System at a glance

DepthMark follows a strict medallion architecture. Bronze is the only filesystem-backed layer — it preserves FotMob API payloads exactly as received. Silver and Gold live entirely in ClickHouse. Gold splits into three independent namespaces so scenarios, signals, and shared serving metadata evolve without coupling. SQL owns all analytical logic; Python discovers jobs, executes, validates, and reports.

Boundary 1

Bronze = source fidelity

Preserve FotMob payloads as-is on disk and in bronze.*. No business logic, no typing, no joins.

Boundary 2

Silver = conformed analytics

Standardize keys, types, and entities. Full-table SQL reloads from the complete Bronze dataset — no incremental watermarks.

Boundary 3

Gold = product outputs

Scenarios and signals materialize business-facing tables. One activation serving table unifies all signals into a single queryable surface.

Ownership

SQL transforms, Python runs

All analytical logic lives in .sql files. Python discovers jobs, executes, validates, and reports. No per-signal Python wrappers.

Idempotency

Safe reruns everywhere

ReplacingMergeTree on Bronze, Silver, Gold signals, and activations. Re-run any stage without manual cleanup or rollback.

Catalogs

Git is truth for metadata

Signal catalogs are markdown frontmatter in Git. MongoDB is a synchronized serving copy — product APIs read from MongoDB, never ClickHouse.

Archives

S3 sync is independent

Bronze upload and download are explicit operator workflows. Scraping and warehouse runs never depend on S3 availability.

Bronze Source fidelity

The only filesystem-backed layer. Scrapes FotMob API payloads to disk sequentially, then parses them into 15 raw ClickHouse tables via Pydantic extraction. Failed inserts land in a dead-letter queue — no data is silently dropped. Bronze has no business logic, typing, or joins; those belong in Silver. S3 archive transfer is an independent operator action and is never part of scraping or the warehouse pipeline.

Deep dive: bronze.md

🌐 Source

FotMob API — one JSON payload per match, fetched sequentially to reduce rate-limit and ban risk

💾 Storage

Historical match files at data/fotmob/historical/matches/YYYYMMDD/, listings at data/fotmob/historical/daily_listings/YYYYMMDD/. Live files remain isolated under data/fotmob/live/. Failed inserts → data/dlq/

♻️ Idempotency

ReplacingMergeTree(inserted_at) — re-running a date safely overwrites via version column

Silver Conformed analytics

Bronze tables are joined, typed, and cleaned into 8 reusable analytical tables. Silver always reloads from the complete Bronze dataset — there is no incremental watermark. Every Silver run is deterministic and safe to re-run. All 344 signals and 48 scenarios read exclusively from these 8 tables, never directly from Bronze.

Deep dive: silver.md

💡 Design rationale: Silver is the stable analytical foundation. Keeping Gold SQL focused on business logic — not source parsing — means you can change FotMob payload structure without touching any of your 392 Gold tables.

Pipeline orchestration

pipeline.py coordinates all layers via subprocess calls (ADR 0002). Any non-zero exit halts the run immediately — there is no partial-success state. Skip and partial-run flags support fast iterative development without rerunning the full pipeline. Bronze S3 sync is deliberately outside this boundary; pipeline success never depends on S3 availability.

Infrastructure

Local development runs via Docker Compose. ClickHouse holds all warehouse layers (bronze, silver, gold_scenarios, gold_signals, gold). MongoDB holds content catalogs in the orbit_content database. Credential policy differs significantly between local and production — never expose ports in production. Optional S3-compatible storage is used only by the standalone Bronze archive sync command.

MongoDB — content serving

MongoDB stores synchronized copies of signal and scenario catalogs for product APIs. Git markdown frontmatter is the single source of truth; MongoDB is never authored directly. The sync script validates required frontmatter keys, checks gold_signals.* namespace compliance, and uses SHA-256 content hashing to make upserts idempotent.

Deep dive: mongodb.md

💡 Key design constraint: Product APIs read from MongoDB, not from ClickHouse. This decouples product query patterns (flexible document queries) from warehouse query patterns (analytical SQL). Never query ClickHouse catalogs from product APIs.
